## Slide 2: UCIe RX-Path Design Under Test (DUT) Overview
* **Design Block:** `rx_path`
* **Core Functionality:**
  * **Link Training State Machine (LTSM) Interface:** Processes training patterns (Clock, Valid, Lane ID, and LFSR sequences) and generates status indications.
  * **Logical-to-Physical (Lane-to-Byte - L2B) Mapping:** Maps 16 physical lanes into the standard RDI (Raw Die Interface) byte array, supporting package degradation modes (x16, x8, x4).
  * **LFSR Descrambling:** Implements 16 independent, parallel modulo-8 LFSR descramblers to reverse partner-die scrambling.
  * **Deskew & Clock Alignment:** Detects and tracks physical training streams.
* **Architectural Parameters:**
  * `pNUM_LANES = 16` (Physical serial mainband lanes)
  * `pDATA_WIDTH = 64` (Bits processed per lane per cycle - UI serialization depth)
  * `pNBYTES = 256` (Total RDI data bus width: 256 Bytes = 2048 bits)
  * `pLFSR_TAPS = 23` (Polynomial tap width)

## Slide 3: Top-Level Testbench Architecture (`rp_tb_top`)
* **Infrastructure Clocks:**
  * `clk`: Sideband/LTSM System Clock (Period = 64ns)
  * `hclk`: High-speed clock (Period = 2ns)
  * `dclk`: Double Data Rate sampling clock (Period = 1ns)
  * `reset_wire`: Global asynchronous reset generated by `rp_reset_intf`
* **Interface Instantiations:**
  * `rp_reset_intf`: Generates controlled reset sequences.
  * `rp_rdi_bfm`: Monitors output RDI bus transaction streams (`pl_data`, `pl_valid`).
  * `rp_ltsmc_bfm`: BFM to drive LTSM states & capture results (`o_rx_done`, `o_rx_data_results`, `o_clk_result`, `o_valid_result`).
  * `rp_rmblink_bfm`: Mainband BFM driving serial differential clock/valid/data lanes.
* **DUT Bind Directive:**
  * SystemVerilog Assertions (`rp_sva`) are bound directly inside `rx_path dut` to verify internal patterns and output timings concurrently.

## Slide 4: UVM Environment (`rp_env`) Topology
* **Structure:**

  * `ltsmc_agent (ltsmc_agt)`: active agent driving LTSM status configurations.
  * `rmblink_agent (rmblink_agt)`: active agent driving serial physical lane stimulus.
  * `virtual_sequencer (vseqr)`: coordinates sequence flow across agents.
  * `rp_scoreboard (sb)`: contains predictor (`rp_pred`) and output comparators (`cmp_rdi`, `cmp_ltsmc`).
  * `rp_coverage_collector (cvg)`: collects state, transition, and configuration coverage.
* **Analysis Connections:**
  * `rmblink_agt.in_ap` $\rightarrow$ Scoreboard & Coverage Collector.
  * `ltsmc_agt.in_ap` $\rightarrow$ Scoreboard & Coverage Collector.
  * `rdi_agt.out_ap` $\rightarrow$ Scoreboard RDI Comparator.
  * `ltsmc_agt.out_ap` $\rightarrow$ Scoreboard LTSMC Comparator.

## Slide 5: LTSM Agent (`ltsmc_agent`) Detail
* **Purpose:** Emulates the Link Training State Machine sideband configurations.
* **BFM Interface Signals:**
  * `i_lane_map_code`: Selects active layout (`X16_MODE=3'b011`, `X8_LOWER_MODE=3'b001`, `X8_UPPER_MODE=3'b010`, `X4_LOWER_MODE=3'b100`, `X4_UPPER_MODE=3'b101`).
  * `i_rx_encoding`: Current state of the RX FSM (9-bit code).
  * `i_error_threshold`: Acceptable bit-error threshold (16-bit) during training.
  * `i_half_rate`: Frequency scaling selector.
  * `o_rx_done`: Indication from DUT that the current training step has finished.
  * `o_rx_data_results`: Per-lane status of LFSR lock.
  * `o_clk_result`: Clock pattern lock status (`[2:0]`: `track`, `clk_n`, `clk_p`).
  * `o_valid_result`: Valid pattern lock status.
* **Sequencing Modes (`ltsmc_sequence`):**
  * `NEXT`: Steps to the subsequent state in the linear training array.
  * `CUSTOM`: Jumps directly to a target encoding (e.g., forcing `RESET_Reset`).
  * `TRAVERSE`: Steps sequentially through intermediate states to reach a target.

## Slide 6: RMBLINK Agent (`rmblink_agent`) Detail
* **Purpose:** Drives serial mainband physical line transactions.
* **Physical Wires driven by BFM:**
  * `i_clk_p`, `i_clk_n`: Serial differential clock lanes.
  * `i_track`: Tracking lane for deskew calibration.
  * `i_data [15:0]`: 16 serial data lanes.
  * `i_valid`: Valid lane representing framing.
* **BFM Methods:**
  * `serialize_data(...)`: Runs at double data rate.
    * Posposedge `i_dclk`: Drives `i_data` and `i_valid`.
    * Negnegedge `i_dclk`: Drives `i_clk_p`, `i_clk_n`, and `i_track`.
  * `serialize_clk_pattern(...)`: Drives clock training sequences.
  * `serialize_valid_pattern(...)`: Drives valid training patterns.
* **Sequence Item Configuration (`rmblink_seq_item`):**
  * Randomizes `val_stream`, `clk_stream_p/n`, `track_stream`, `data` array, `idle_ui_cnt` insertion, and `rp_opmode` (`CLK_PATTERN`, `VAL_PATTERN`, `DATA_PATTERN`, `ACTIVE`).

---

## Slide 7: RDI Agent (`rdi_agent`) Detail
* **Purpose:** Monitors the Raw Die Interface (RDI) boundary at the adapter layer.
* **Interface Signals monitored by BFM:**
  * `pl_data [255:0][7:0]`: Reconstructed, mapped, and descrambled parallel data byte array (2048 bits).
  * `pl_valid`: Valid packet framing signal presented to the Adapter layer.
* **UVM Transaction (`rdi_seq_item`):**
  * Captures the massive 256-Byte packet payload.
  * In active mode, RDI transactions are captured upon the assertion of `pl_valid` and sent to the scoreboard for comparison.

## Slide 8: UVM Scoreboard & Predictor Modeling Logic
* **Predictor (`rp_pred`) Golden Models:**
  * **LFSR Descrambling Model:** Modulo-8 parallel implementation matching the tap feedback logic:
    * Tap indices [22] (Feedback), [21], [16], [8], [5], [2] are XORed during shifts:
    $$\text{State}[j] = \text{State}[j-1] \oplus \text{State}[22] \quad (\text{for } j \in \{2, 5, 8, 16, 21\})$$
    $$\text{State}[0] = \text{State}[22]$$
    * XORs physical data lanes with the computed LFSR stream to generate descrambled output.
  * **Lane-to-Byte (L2B) Mapping Model:** Reconstructs the 256-Byte RDI packet by placing incoming lane bits into RDI bytes based on the active lane map code:
    $$\text{RDI\_Byte}[\text{step} \times (8 \times \text{iteration} + \text{byte\_idx}) + \text{norm\_idx}] = \text{Lane\_Data}[\text{lane}][\text{byte\_idx}]$$
  * **Per-Lane ID Detection Model:** Evaluates logical-to-physical mapping. Expects a repeating 16-bit word per lane containing the physical index:
    $$\text{Expected\_Word} = \{\text{4'b1010}, \text{8'd(lane\_index)}, \text{4'b1010}\}$$
    Locks logic lane when 16 consecutive matches occur without exceeding the error threshold.

## Slide 9: Verification Plan - Reset and Training Scenarios
* **RX_1: RX-Controller Reset & Idle**
  * **Stimulus:** Asserts global reset or `RESET` state. Drives idle patterns on RMBLINK.
  * **Checking:** Scoreboard and SVA verify `o_rx_done`, `o_clk_results`, `o_valid_results`, `o_pl_valid`, and `o_pl_data` are cleared.
* **RX_2: Clock Pattern Detection (Pass)**
  * **Stimulus:** alternating 1/0 clock stream driven on physical clock lanes during `CLK_TRAIN` encoding.
  * **Checking:** SVA verifies `o_clk_results` asserts after exactly 16 consecutive valid 48-bit windows.
* **RX_3: Clock Pattern Detection (Error)**
  * **Stimulus:** Corrupted clock stream with bit-flips injected during training.
  * **Checking:** Verifies that consecutive window counters reset and `o_clk_results` remains 0.
* **RX_4: Valid Pattern Detection**
  * **Stimulus:** 8'b1111_0000 framing patterns driven on `i_valid` with randomized `i_error_threshold`.
  * **Checking:** SVA verifies lock if error counts are $\le$ threshold, and fails if they exceed it.

## Slide 10: Verification Plan - Lane ID & Active Data Scenarios
* **RX_5: Lane ID Detection (MBINIT)**
  * **Stimulus:** Logical lane index patterns driven on active lanes during `MBINIT` reversal.
  * **Checking:** Predictor verifies logical lane IDs are detected and status asserts per lane after 16 matches.
* **RX_6: Lane ID Detection (Error)**
  * **Stimulus:** Corrupted lane ID payload injected before the 16th match.
  * **Checking:** Scoreboard verifies that output status bits remain low for corrupted lanes.
* **RX_7: LFSR Descrambling (ACTIVE)**
  * **Stimulus:** Scrambled random data driven on RMBLINK during `ACTIVE` state.
  * **Checking:** Scoreboard verifies descrambled RDI output exactly matches the unscrambled reference.
* **RX_8: x16 Lane-to-Byte Mapping**
  * **Stimulus:** All 16 lanes active. Drives randomized payload.
  * **Checking:** Scoreboard checks that physical Lane $N$ maps directly to RDI Byte $N$ ($N: 0 \text{ to } 15$).

## Slide 11: Verification Plan - Width Degradation & Valid Framing
* **RX_9: Width Degrade: x8 Mode**
  * **Stimulus:** Maps active lanes to `X8_LOWER_MODE` (Lanes 0-7) or `X8_UPPER_MODE` (Lanes 8-15).
  * **Checking:** Scoreboard verifies that active lanes are mapped to the lower bytes of `o_pl_data` and inactive lanes are ignored (High-Z).
* **RX_10: Width Degrade: x4 Mode**
  * **Stimulus:** Maps active lanes to `X4_LOWER_MODE` (Lanes 0-3) or `X4_UPPER_MODE` (Lanes 4-7).
  * **Checking:** Scoreboard checks correct sequential mapping onto RDI byte segments.
* **RX_11: RDI Valid Framing**
  * **Stimulus:** Introduces periodic framing gaps (valid drops to 0) on the serial interface.
  * **Checking:** Scoreboard verifies that `o_pl_valid` follows data framing precisely with zero phase lag.

## Slide 12: Detailed Test Scenarios - Valid Pattern Sequence (`rmblink_sanity_valid_sequence`)
* **Test Configurations (Directed & Random):**
  * `TEST_PURE_RANDOM`: Valid and clock lanes driven with pure noise.
  * `TEST_IDEAL_ALL_0F`: Perfect repeating `8'b1111_0000` sequence.
  * `TEST_INJECT_START`: Injects exactly 16 valid patterns at the beginning of the stream.
  * `TEST_INJECT_MIDDLE`: Injects exactly 16 valid patterns starting at cycle 32.
  * `TEST_INJECT_END`: Injects exactly 16 valid patterns at the end of the stream.
  * `TEST_SINGLE_ERROR` (Start/End/Random): Injects a single bit flip in the pattern. Verifies lock recovery when threshold is set $\ge 1$.
  * `TEST_MULTI_ERR_ABOVE_THRESH`: Injects multiple errors exceeding the threshold. Verifies lock failure.
  * `TEST_ACTIVE_IDLE`: Simulates inter-packet transitions.
  * `TEST_IDEAL_VALID_RANDOM_CLKS`: Perfect valid stream, but clocks are corrupted. Verifies that valid lock fails if clocks are unstable.

---

## Slide 13: Detailed Test Scenarios - Lane ID Sequence (`rmblink_sanity_PerLaneID_sequence`)
* **Stress Test Scenarios for Lane IDs:**
  * `SCENARIO_IDEAL`: Ideal repeating index patterns driven for 32 iterations (128 words).
  * `SCENARIO_FAIL_MIDWAY`: Injects a single corrupted payload at cycle 10. Verifies lock failure.
  * `SCENARIO_NOISE_THEN_IDEAL`: Sends random noise for the first 15 cycles, then ideal patterns. Verifies delayed lock.
  * `SCENARIO_WRONG_LANE_ID`: Valid pattern framing, but lane payload is shifted (Lane $N$ drives index $N+1$). Verifies failure.
  * `SCENARIO_RANDOM_INTERLEAVED`: Randomly alternates between valid lane IDs and garbage.
  * `SCENARIO_MIXED_SUCCESS`: Even lanes drive correct IDs, odd lanes drive garbage. Verifies that only even lanes assert lock.
  * `SCENARIO_LATE_SUCCESS`: Valid patterns driven only during the last 16 cycles. Verifies successful delayed lock.
  * `SCENARIO_THRESHOLD_TEASER`: Sends exactly 15 valid words, then 1 garbage word, and repeats. Verifies that the internal accumulation counter is reset and lock is never achieved.

## Slide 14: SystemVerilog Assertions (SVA) Infrastructure
* **Role:** Concurrently monitors physical interface protocol rules and checks timing.
* **Key Properties and Assertions:**
  * `assert_valid_pattern_16_frame`: Checks that a valid lock occurs if and only if 16 consecutive `8'b11110000` frames are detected on `i_valid`.
    * Implemented via SVA sequences:
      ```systemverilog
      sequence seq_8bit_pattern;
          i_valid[*4] ##1 (!i_valid)[*4];
      endsequence
      sequence seq_16_consecutive;
          seq_8bit_pattern[*16];
      endsequence
      ```
  * `assert_clk_p_pattern_frame`, `assert_clk_n_pattern_frame`, `assert_track_pattern_frame`: Checks clock training lock conditions on physical lines.
  * `assert_valid_pattern_result_check` & `assert_clk_pattern_result_check`: Compares lock outputs (`o_valid_results`, `o_clk_results`) against SVA internal tracking flags at state transition handshakes.
  * `chk_async_reset_zeros` & `chk_async_reset_ones`: Verifies that outputs transition immediately to default values upon reset.

## Slide 15: Functional Coverage Model
* **LTSM State Coverage (`cp_encoding`):**
  * Individual bins covering every SystemVerilog state encoding for SBINIT, MBINIT, training sweeps, active, low-power sleep (L1), and error modes.
* **Lane Map Coverage (`cp_lane_map`):**
  * Bins for x16, x8 (upper/lower), and x4 (upper/lower) mapping configurations.
* **Cross Coverage (`cx_state_x_lane_map`):**
  * Crosses lane maps with width degradation states (`ST_REPAIRMB` and `ST_REPAIR`).
* **LTSM Transition Coverage (`cp_transitions`):**
  * Bins for all legal state transitions (happy path, retrain paths, low power wakeup, and error exit branches).
* **Cross lock coverage:**
  * Bins checking valid and clock lock result status registers at handshake states.
